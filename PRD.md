# Mini PRD: 점심 메뉴 추천 서비스 '점메추'

---

## 0. PRD 한 줄 요약 (One-liner)

이 서비스는 **메뉴 고민하기 싫은 직장인**이 현재 위치 주변의 음식점을 고민 없이 선택하도록 돕는
**위치 기반 점심 메뉴 추천 웹 서비스**이다.

---

## 1. Problem (문제 정의)

### 1-1. 어떤 문제가 있는가?

- **상황**: 점심시간마다 "오늘 뭐 먹지?"를 반복하며 시간을 낭비하고, 결국 매번 비슷한 곳만 가게 됨.
- **중요성**: 짧은 점심시간에 메뉴 결정에 에너지를 소모하면 오후 업무 집중도가 저하됨.
- **가이드 답변**: 결정 직전에 발생하는 문제로, 현재는 동료에게 물어보거나 지도·배달앱을 뒤지는 방식에 의존함. 선택지가 너무 많아 결정 피로도만 증가함.

### 1-2. 왜 아직 잘 해결되지 않았는가?

- **기존 해결책의 한계**: 배달앱은 배달 위주라 직접 방문 식당 탐색에 불편하고, 지도 앱은 정보가 너무 많아 오히려 결정을 어렵게 만듦. 개인의 **비선호 음식을 자동으로 걸러주는** 기능이 없어 매번 스스로 필터링해야 함.

---

## 2. Target User (타겟 사용자)

### 2-1. 핵심 사용자 1명 정의

- **누구**: 30대 직장인 (회사 근처 식당에서 점심을 먹는 사람)
- **맥락**: 점심시간 1시간 중 메뉴 고르는 데만 10~15분을 쓰는 사람. 빠르게 결정하고 싶지만 매번 같은 곳 가기는 싫고 새로운 곳도 가보고 싶음.

### 2-2. 사용 시나리오 (짧게)

점심시간 직전, 회사에서 서비스에 접속. 현재 위치를 허용하고 가격대와 비선호 음식(예: 매운 음식, 날 음식)을 선택하면,
근처 음식점 3곳이 카드 형태로 추천됨. 고민 없이 하나를 골라 바로 출발.

---

## 3. Solution (해결책)

### 3-1. 우리가 제공하는 핵심 해결 방식

- **핵심**: "메뉴 결정의 외주화(Outsourcing)"
- **딱 하나만 말하면**: 현재 위치와 내 취향(비선호 음식 필터)을 기반으로 **'지금 당장 갈 수 있는 최선의 음식점 3곳'** 을 딱 정해준다.

### 3-2. 핵심 기능 (3개 이내)

1. **위치 기반 음식점 탐색**
   - GPS로 현재 위치를 감지하여 사용자가 선택한 반경(500m / 1km / 2km) 내 음식점을 Google Maps API로 검색
   - Google Maps 리뷰에 기재된 **1인당 가격 정보**를 가격대 필터로 활용

2. **비선호 음식 필터링**
   - 최초 접속 시 또는 설정 화면에서 싫어하거나 못 먹는 음식 카테고리를 선택
   - 추천 단계에서 **비선호 음식만 판매하는 식당을 자동 제외**하고 결과 제공
   - (예: '매운 음식' 비선호 → 마라탕·불닭 전문점 제외)

3. **AI 기반 메뉴 정보 보강**
   - 생성형 AI API(예: Claude / GPT)를 활용해 각 음식점의 **대표 메뉴 정보**를 검색·보완
   - 음식점명, 카테고리, 대표 메뉴, 가격대, 거리 정보를 카드 형태로 제공

---

## 4. User Flow (아주 간단한 흐름)

```
1. 사이트 접속
   └─ 최초 방문: 로그인(Firebase Auth) 후 비선호 음식 카테고리 선택 (DB 저장, LocalStorage는 캐시)

2. 조건 선택
   └─ 탐색 반경 선택 (500m / 1km / 2km)
   └─ 가격대 선택 (1만원 이하 / 1~1.5만원 / 1.5만원 이상)

3. 결과 확인
   └─ 비선호 음식 필터 적용 후 추천 음식점 3곳 카드 표시
   └─ 카드 정보: 음식점명 · 카테고리 · 대표 메뉴 · 가격대 · 거리

4. 선택
   └─ 카드 선택 → Google Maps로 연결 (경로 안내)
```

---

## 5. Success Metric (성공 기준)

- 서비스 접속 후 추천 카드까지 도달하는 **전환율 80% 이상**
- 추천된 3개 카드 중 1개를 선택하는 **선택 완료율 60% 이상**

---

## 6. 음식 카테고리 정의

비선호 음식 필터 및 식당 분류에 사용되는 카테고리.

| 대분류 | 세부 항목 (예시) | 비선호 필터 태그 예시 |
|--------|----------------|----------------------|
| 한식 | 백반, 국밥, 냉면, 쌈밥, 갈비/삼겹살 | 매운 음식, 내장류 |
| 중식 | 짜장면, 짬뽕, 마라탕, 탕수육 | 마라/향신료, 기름진 음식 |
| 일식 | 초밥/회, 라멘, 돈카츠, 우동 | 날 음식(생선/회) |
| 양식 | 파스타, 스테이크, 샌드위치, 버거 | 유제품, 밀가루 |
| 분식 | 떡볶이, 김밥, 순대, 튀김 | 매운 음식, 밀가루 |
| 아시안 | 쌀국수, 태국음식, 인도음식 | 고수, 향신료 |
| 패스트푸드 | 햄버거, 치킨 프랜차이즈 | 패스트푸드 전체 |
| 카페/브런치 | 브런치, 샐러드, 카페 밥 | (필터 대상 낮음) |

> **비선호 필터 태그 예시**: 매운 음식 / 날 음식(생선·회) / 고수 / 내장류 / 마라·향신료 / 유제품 / 밀가루(글루텐) / 돼지고기 / 해산물

## 7. Scope & Constraints (범위 제한)

### 7-1. 이번 프로젝트에서 하지 않는 것

- **로그인 시스템**: Firebase Authentication — Google 소셜 로그인만 제공 (게스트 모드 없음)
- **예약/주문 연동**: 추천까지만 제공, 결제·예약은 Google Maps 등 외부 서비스로 연결
- **모바일 앱**: 웹 서비스 형태에 집중 (모바일 반응형 웹으로 대응)

### 7-2. 현실적인 제약

- **API 비용**: Google Maps API는 호출 횟수당 비용 발생 → 결과 캐싱 전략 필요
- **가격 데이터 누락**: Google Maps에 1인당 가격 정보가 없는 식당은 '가격 정보 없음'으로 표시 후 포함
- **필터 정확도**: AI 카테고리 분류의 정확도에 따라 필터 결과가 달라질 수 있음
- **개발 기간**: 소규모 팀으로 단기 내 핵심 기능(MVP) 구현 목표

---

## 8. Tech Stack (기술 스택)

| 역할 | 사용 기술 |
|------|----------|
| Frontend | Next.js (React) + TypeScript |
| Hosting/Infra | AWS CloudFront + S3 (프론트) + Lambda + API Gateway (백엔드) |
| Authentication | Firebase Authentication |
| 위치 정보 | Browser Geolocation API |
| 음식점 검색 | Google Maps Places API |
| 가격 정보 | Google Maps Places API (price_level / reviews) |
| 메뉴 정보 보강 | 생성형 AI API (Claude / GPT) |
| 데이터 저장 | AWS RDS PostgreSQL + Redis (ElastiCache), LocalStorage는 캐시/임시 저장 |
| 외부 연결 | Google Maps URL Scheme (경로 안내) |


---

## 9. Why Now? (왜 지금인가)

- **시장 트렌드**: '결정 피로도'와 '선택 과부하'가 현대인의 주요 스트레스 원인으로 부각되는 가운데, 위치 기반 서비스와 AI 추천의 결합으로 **초개인화된 오프라인 식당 추천**이 기술적으로 충분히 실현 가능한 시점임.
- **API 접근성**: Google Maps API와 생성형 AI API의 가격 장벽이 낮아져 소규모 팀도 빠르게 MVP 구현 가능.
